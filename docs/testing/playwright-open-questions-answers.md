# Playwright Open Questions Answers

## SECTION A ‚Äî JWT PAYLOAD & PERSONA TRUTH
A1) ‚úÖ Confirmed ‚Äì `JwtStrategy.validate()` returns `{ userId, email, mustChangePassword, role, isAdmin }` and that object becomes `req.user`. Evidence: `apps/api/src/auth/jwt.strategy.ts: validate()`.
A2) ‚úÖ Confirmed ‚Äì `AdminGuard` inspects `request.user?.isAdmin` and throws `ForbiddenException` if the flag is falsy, so only users with `isAdmin` succeed. Evidence: `apps/api/src/auth/auth.guard.ts: AdminGuard`.
A3) ‚úÖ Confirmed ‚Äì there is no RBAC beyond the `role ('user'|'admin')`/`isAdmin` pair, the only custom guard is `AdminGuard`, and the web app only renders admin nav/pages when `auth.me.isAdmin` is true. Evidence: `apps/api/src/db/schema.ts: role/isAdmin`, `apps/api/src/auth/auth.guard.ts: AdminGuard`, `apps/web/app/hooks/useAuth.ts: Me`, `apps/web/app/components/Layout.tsx: admin links gated on isAdmin`.

## SECTION B ‚Äî CSRF MECHANICS (CRITICAL)
B1) ‚úÖ Confirmed ‚Äì `CsrfGuard` requires `x-csrf-token` (default) and the `todo_csrf` cookie, enforces non-GET/HEAD/OPTIONS methods only, and compares the two tokens with `timingSafeEqual`. Evidence: `apps/api/src/common/csrf.ts: CSRF_HEADER_NAME`, `createCsrfToken()`, `setCsrfCookie()`, `CsrfGuard.canActivate()`.
B2) ‚úÖ Confirmed ‚Äì `CsrfGuard` is applied wherever state-changing auth requests occur (`logout`, `change-password`, `me` in `apps/api/src/auth/auth.controller.ts`) and at the class level for `attachments`, `categories`, `todos`, `remarks`, `settings`, `audit`, and `admin` controllers (all decorated with `@UseGuards(JwtAuthGuard, CsrfGuard, ...)`). Evidence: `apps/api/src/auth/auth.controller.ts`, `apps/api/src/attachments/attachments.controller.ts`, `apps/api/src/categories/categories.controller.ts`, `apps/api/src/todos/todos.controller.ts`, `apps/api/src/remarks/remarks.controller.ts`, `apps/api/src/settings/settings.controller.ts`, `apps/api/src/audit/audit.controller.ts`, `apps/api/src/admin/admin.controller.ts`.
B3) ‚úÖ Confirmed ‚Äì `setCsrfCookie()` issues the cookie with `httpOnly: false`, `path: '/'`, sameSite/secure derived from `COOKIE_SAMESITE`/`COOKIE_SECURE`, making the token readable by JavaScript. Evidence: `apps/api/src/common/csrf.ts: setCsrfCookie()`.
B4) ‚úÖ Confirmed ‚Äì the frontend reads `todo_csrf` via `document.cookie`, sets the header before every `apiFetchJson` call, and sends `credentials: 'include'` so the backend double-submit token check is satisfied. Evidence: `apps/web/app/lib/api.ts: getCsrfToken()`, `applyCsrfHeader()`, `apiFetchJson()`.

## SECTION C ‚Äî REGISTER / LOGIN UI REALITY
C1) ‚úÖ Confirmed ‚Äì the root `app/page.tsx` renders `LoginForm`, which toggles between login/register modes via the ‚ÄúDon‚Äôt have an account? Register‚Äù button, exposing the register flow on `/`. Evidence: `apps/web/app/page.tsx: Home` and `apps/web/app/components/LoginForm.tsx` (toggle logic and register UI).
C2) ‚úÖ Confirmed ‚Äì `useAuth.register()` POSTs to `/auth/register`, immediately POSTs to `/auth/login`, then calls `refreshMe()` so the UI relies on the backend cookie plus the login call before showing authenticated content. Evidence: `apps/web/app/hooks/useAuth.ts: register()`, `apps/web/app/lib/api.ts: apiFetchJson()` (`credentials: 'include'`).
C3) ‚úÖ Confirmed ‚Äì once `auth.me` exists and no forced password change is needed, `Home` (`apps/web/app/page.tsx`) renders the task dashboard at `/` (My Tasks). Evidence: `apps/web/app/page.tsx: renders tasks/Layout when auth.me is truthy`.
C4) ‚úÖ Confirmed ‚Äì `ForcePasswordChange` is rendered whenever `me.mustChangePassword` is true on the home page or admin/activity/customizations pages, so any user flagged by the backend (bootstrap admin, reset password) must change before seeing other UI. Evidence: `apps/web/app/components/ForcePasswordChange.tsx` plus the checks in `apps/web/app/page.tsx`, `apps/web/app/admin/page.tsx`, `apps/web/app/activity/page.tsx`, `apps/web/app/customizations/page.tsx`.

## SECTION D ‚Äî ADMIN BOOTSTRAP & PROMOTION
D1) ‚úÖ Confirmed ‚Äì `BootstrapService.onModuleInit()` runs on startup unless `SKIP_BOOTSTRAP=true`, then it ensures default system settings and the admin user. Evidence: `apps/api/src/bootstrap/bootstrap.service.ts: onModuleInit()`.
D2) ‚úÖ Confirmed ‚Äì defaults are `DEFAULT_ADMIN_EMAIL = 'admin@example.com'`/`DEFAULT_ADMIN_PASSWORD = 'admin123'`, but `ConfigService` prefers `ADMIN_EMAIL`/`ADMIN_PASSWORD` from the environment. Evidence: `apps/api/src/bootstrap/bootstrap.service.ts: DEFAULT_ADMIN_*` and `this.config.get('ADMIN_EMAIL'/'ADMIN_PASSWORD')`.
D3) ‚úÖ Confirmed ‚Äì when a new admin is created the service logs the email and password with `Logger.log()` before exiting so the credentials are printed to the console. Evidence: `apps/api/src/bootstrap/bootstrap.service.ts: this.logger.log('Created admin user...', 'Default password: ...')`.
D4) ‚úÖ Confirmed ‚Äì bootstrapped admins are saved with `mustChangePassword: true`, and the frontend immediately shows `ForcePasswordChange` until the flag clears, so they must change before accessing the app. Evidence: `apps/api/src/bootstrap/bootstrap.service.ts: insert users with mustChangePassword: true`, `apps/web/app/components/ForcePasswordChange.tsx` and its usage (see Section C4).
D5) ‚úÖ Confirmed ‚Äì promoting/demoting admins is done via `POST /admin/users/:id/toggle-admin` (guarded by `JwtAuthGuard`, `CsrfGuard`, `AdminGuard`) which calls `AdminService.toggleAdmin()` to flip `isAdmin` and `role`. Evidence: `apps/api/src/admin/admin.controller.ts: toggleAdmin()` and `apps/api/src/admin/admin.service.ts: toggleAdmin()`.

## SECTION E ‚Äî UI ROUTES VS BACKEND ENFORCEMENT
E1) ‚úÖ Confirmed ‚Äì `/admin`, `/activity`, `/customizations` pages each check `me?.isAdmin` and immediately redirect `window.location.href = '/'` when false, so non-admin users are bounced to the home dashboard. Evidence: `apps/web/app/admin/page.tsx`, `apps/web/app/activity/page.tsx`, `apps/web/app/customizations/page.tsx` (the `if (!me.isAdmin)` blocks).
E2) ‚úÖ Confirmed ‚Äì admin-only endpoints and their guards are:
- `GET /admin/users`, `POST /admin/users/:id/reset-password`, `POST /admin/users/:id/toggle-admin` (class `@UseGuards(JwtAuthGuard, CsrfGuard, AdminGuard)`).
- `GET /audit` and `GET /audit/all` (each annotated with `@UseGuards(JwtAuthGuard, CsrfGuard, AdminGuard)`).
- `POST /categories`, `PUT /categories/:id`, `DELETE /categories/:id`, `POST /categories/seed-defaults` (each using `@UseGuards(AdminGuard)` on top of the Jwt+Csrf controller).
- `PUT /settings`, `PUT /settings/duration` (AdminGuard on both).
Evidence: `apps/api/src/admin/admin.controller.ts`, `apps/api/src/audit/audit.controller.ts`, `apps/api/src/categories/categories.controller.ts`, `apps/api/src/settings/settings.controller.ts`.
E3) ‚úÖ Confirmed ‚Äì `GET /audit/resource/:resourceId` is guarded by `JwtAuthGuard`+`CsrfGuard`, and `AuditService.getResourceHistory()` enforces that only the owning todo‚Äôs user (or an admin) can view its logs. Evidence: `apps/api/src/audit/audit.controller.ts: getResourceHistory()` and `apps/api/src/audit/audit.service.ts: getResourceHistory()`.
E4) ‚úÖ Confirmed ‚Äì every admin API path is explicitly decorated with `AdminGuard`, which throws `ForbiddenException` for non-admins, so there are no UI-hidden admin APIs that standard users can call. Evidence: `apps/api/src/auth/auth.guard.ts: AdminGuard` plus the annotations listed in E2.

## SECTION H ‚Äî CALENDAR & OCR STABILITY
H1) ‚úÖ Confirmed ‚Äì scheduling can be done via the `ScheduleModal` form (start time + duration) that is opened from the task list or calendar‚Äîno drag/drop needed. Evidence: `apps/web/app/page.tsx: ScheduleModal` usage, `apps/web/app/calendar/page.tsx: openScheduleModal()` and the `ScheduleModal` component itself.
H2) ‚úÖ Confirmed ‚Äì drag/drop relies on `DraggableTask` (`useDraggable` with id `draggable-${taskId}`) plus `DroppableZone` IDs `calendar-grid` and `unschedule-zone`, and `DragProvider` uses those IDs to know whether to schedule/unschedule/reschedule. Evidence: `apps/web/app/components/DraggableTask.tsx`, `apps/web/app/components/DroppableZone.tsx`, `apps/web/app/components/DragContext.tsx`, `apps/web/app/calendar/page.tsx` (zones `id="calendar-grid"` and `id="unschedule-zone"`).
H3) ‚úÖ Confirmed ‚Äì OCR runs via a standalone worker (`apps/ocr-worker/main.py`) that exposes `POST /ocr`, and the API calls it through `OcrService.extractFromWorker()` which POSTs to `${OCR_WORKER_BASE_URL}/ocr`. Evidence: `apps/api/src/ocr/ocr.service.ts: extractFromWorker()`, `apps/api/src/attachments/attachments.controller.ts: triggerOcr()`, `apps/ocr-worker/main.py: POST /ocr`.
H4) ‚úÖ Confirmed ‚Äì the Docker Compose stack already builds/starts the `ocr-worker` service and `.env`/docs point `OCR_WORKER_BASE_URL` to it, so OCR is available when running `docker compose up --build`. Evidence: `docker-compose.yml: ocr-worker service`, `.env: OCR_WORKER_BASE_URL`, `docs/testing/playwright-setup-recommendations.md: "OCR worker... must be reachable"`.
H5) ‚úÖ Confirmed ‚Äì given the extra container, PaddleOCR/python dependencies, and dedicated worker noted in the docs, OCR is heavyweight and should live in regression suites rather than critical smoke runs. Evidence: `apps/ocr-worker/main.py` (PaddleOCR/pdf processing work per request) and `docs/testing/playwright-setup-recommendations.md: "OCR worker... must be reachable"`.
